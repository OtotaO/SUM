## 1. Intent

**Build SUM into the trust substrate for AI-transformed knowledge — verifiable about what changed, lossless where it can be lossless, honest about what isn't proven — and earn the right to be the standard by being the first thing that actually works.**

Three operative phrases:

- *Trust substrate* — not a writing app, not a summarizer; the layer underneath. Apps consume it; standards reference it; agents pass it.
- *AI-transformed knowledge* — every form a piece of knowledge can take when an AI touches it: extraction, distillation, composition, rendering, translation between modalities. The dream of bi-directional tags↔tomes was the seed; the larger frame is movement of knowledge between forms.
- *Honest about what isn't proven* — the proof-boundary discipline is not internal hygiene, but it is **table stakes for the skeptic niche, not the moat**: honesty has no IP, no network effect, no switching cost — a competitor can add a disclaimer in a weekend. The actual (narrow, possibly-defensible) differentiator is the **paraphrase-robust, signed, conformal meaning-loss delta** — precisely what AEX and the detection literature disclaim. Other systems will overclaim and SUM doesn't; that earns *trust*, which is necessary but not sufficient. *(Reframed 2026-06-16 per the fresh-eyes audit: "honesty IS the moat" conflated a necessary property with a defensible advantage, and the only evidence for the "moat" framing was the contaminated adoption sims.)*

## 2. The Why

Three layers, each sufficient on its own to justify continuing when the others falter:

### 2.1 The dream

SUM was never an opportunity calculation. The user has held the bi-directional tags↔tomes vision since before SUM had a name — knowledge should be reshapeable across forms without loss, and the reshaping should be inspectable. The slider is that dream made code. The substrate is the dream made portable. Memory `user_origin_dream` is the durable record. **Without the dream, the project doesn't exist. With the dream, every grant decision is downstream.**

### 2.2 The civilizational stake

AI-generated and AI-transformed content is going to dominate the information landscape within the decade. Without an auditable trust layer, the distinction between *what an author claimed* and *what a model produced* dissolves. With one, AI can amplify rather than corrode human knowledge. SUM is a bet on which auditable trust layer wins.

The bet is concrete: cryptographic receipts + bounded semantic preservation + cross-runtime verifiability + explicit epistemic discipline. The bet may lose to a competitor with the same primitives and better adoption. That's acceptable. The bet would only become net-corrosive if SUM marketed proofs it doesn't have — which is exactly what the proof-boundary discipline prevents.

### 2.3 The personal

The user is a sole founder running this from Brooklyn. The grants are optionality, not validation. The Why must survive the case where no grant lands. It does — because the dream and the civilizational stake are independent of funding, and because the substrate is shipped, the work has already produced durable artifacts (working code, public preprint material, the proof-boundary discipline) that exist regardless of what the next twelve months hold.

## 3. Strategy

**Substrate first → adoption through writers → standards via real-customer pull → product-company shape only if a funder pays for it.**

Expanded:

1. **Substrate first** — done. Render receipts, transform receipts, three-runtime verifiers, cross-runtime fixture K-matrix, evidence-chain wiring, replay-defense window, six-regime compliance validators. The substrate is no longer the bottleneck.
2. **Adoption through writers** — the wedge ICP. Freelance journalists with citation-tracking pain. Academic survey writers needing provenance back to source PDFs. Newsletter writers experimenting with audience-slider variants. Bottom-up, one-person-reachable, no enterprise sales motion required. NOT "regulated-domain enterprise" — that's the second wedge once the first one closes, not the first wedge.
3. **Standards via real-customer pull** — C2PA / W3C / IETF move on anchor-customer adoption. Standards work before customer pull is vaporware. Standards work after customer pull is leverage.
4. **Product-company shape only if a funder pays for it** — the deliberation artifact's central call. The user does not commit to product-company UX / support / sales motion without a grant that buys that runway. If no grant funds it, the path is services-with-trap-warning OR contract day-work + slow-burn dream work.

Parallel threads (none of which are the trunk):

- **Honesty discipline** runs continuously: every claim tagged with epistemic status, no "guarantee" without a same-commit receipt, bench-hardening plan committed to as the path that closes single-step into multi-stage claims.
- **Dogfood** runs continuously: the user uses SUM on their own writing, treats it as primary truth-source about what's missing.
- **Operator hygiene** runs continuously: probes, key rotation, JWKS pinning, audit-script maintenance.

## 5. Success criteria

The hardest part of strategy work. Without criteria, every move feels like progress.

### 5.1 True signals — these matter

| Signal | Why it matters |
|---|---|
| **Any grant lands** | Strategic clarity unlocks; decision-tree branch fires; runway shape determined |
| **Dogfood session produces a falsifiable finding** | Either "the slider works for me" (continue path) or "the slider is missing X" (specific deficit named) |
| **External pull** | Someone outside the project asks for one of the named concepts (Perspective Receipts, Trust Profiles, Epistemic Nutrition Label, receipt-passing, verify --explain, source-chain coverage) — that's a real demand signal |
| **Standards-body inbound** | C2PA / W3C / IETF inquires about SUM — standards path becomes live |
| **First paying customer at $30/mo or $300/yr** | Product-company shape validated empirically, not by deliberation |
| **A journalist or academic survey writer reports they shipped a piece using SUM** | The wedge ICP closes |
| **An enterprise inquiry for compliance-grade provenance** | Services path becomes live |

### 5.2 False signals — these are the auto-pivot trap

| Signal | Why it doesn't matter |
|---|---|
| More substrate PRs merged | Velocity ≠ progress. Substrate is already shipped enough for the wedge |
| More tests passing | Pre-push is already 6/6. Extra green is not extra signal |
| Internal feeling of momentum | The 9-PRs-in-4-days episode from 2026-05-12 to 2026-05-16 felt productive; the standing direction at the time was "wait." The dissonance is the warning |
| Larger model writes a vision document | Helpful for framing. Does not validate adoption |
| Repo gets more stars | Not the wedge |
| Twitter / Hacker News attention | Not the wedge unless it converts to dogfood or pull |
| Speculation about which grant might land | Wait for the signal; don't pre-imagine the branch |
| Standards work in advance of customers | Vaporware-shaped |

The discipline: when a signal arrives, ask whether it's in column A or column B before responding.

## 6. Constraints

Things we explicitly will NOT do:

1. **No product-company commitment without funder signal.** The 2026-05-14 deliberation deferred 4–12 weeks. Even if it feels right, the deferral holds until a grant pays for that shape.
2. **No auto-pivot to substrate work during wait periods.** The 9-PRs-in-4-days episode is the example. Substrate hygiene is fine; substrate shipping to fill time isn't.
3. **No "guarantee" language without a same-commit benchmark receipt.** Bench-hardening discipline. Empirical-benchmark is the strongest status without a proof citation.
4. **No quantum-vocabulary imports.** Bench-hardening doc's non-goals list. Only the engineering techniques transfer.
5. **No cross-repo edits.** Anything under `~/SUMequities` is out of scope.
6. **No consulting engagement without a product-pivot trigger written into the contract.** Services-path trap-warning from the deliberation artifact.
7. **No more than one substrate PR per week** unless pulled by named customer or funder signal. The cadence-discipline lesson from the velocity audit.
8. **No claims about modalities not yet built.** "Universal compression" remains scoped to "semantic compression of modalities with extractable structure."
9. **No silent JWKS removal.** Operator-hygiene constraint from the receipt-format spec.
10. **No commits on behalf of the user without explicit authorization scope.** Per the user's global instruction.

## 7. The loop

The operational cadence. Without this, the charter is theatre.

### 7.1 Per-session (every time a Claude Code session opens on this repo)

1. **Read this charter.** It is the compass.
2. Check `MEMORY.md` for new entries since last session.
3. Check `git log --oneline -10` for what's shipped since last session.
4. Check `gh pr list --state open` for in-flight work.
5. Check `gh run list --branch main --limit 3` for CI state.
6. **If a grant signal arrived since last session:** execute the relevant branch of `PRODUCT_DELIBERATION_2026-05-14.md` §4 decision tree.
7. **If a dogfood finding arrived since last session:** capture, triage against `ZENITH_FRAMING_2026-05-16.md` concepts.
8. **If neither:** default to standing direction (wait + dogfood). Do not auto-pivot.

### 7.2 Weekly (light touch — every Friday or equivalent low-noise moment)

1. Scan for grant signals not yet processed.
2. Check whether the user has dogfooded this week. If no signal of intent, nudge once, drop.
3. Check for external pull (GitHub issues, DMs, conversations). Triage by §5.1 vs §5.2.
4. Re-check the success-criteria table for any column-A signal that should have triggered a response.
5. **Do nothing if nothing in 1-4.** Explicitly note "no signal this week."

### 7.3 On grant signal

1. Read the grant-outcome decision tree in `PRODUCT_DELIBERATION_2026-05-14.md` §4.
2. Execute the branch.
3. Update memory (`project_grant_funnel.md` + `project_direction_2026-05-11.md` if direction changes).
4. If direction changes materially: amend this charter §3 (Strategy) and §4 (Objectives).

### 7.4 On dogfood finding

1. Capture the finding verbatim.
2. Read `ZENITH_FRAMING_2026-05-16.md` — dogfood findings usually invoke Perspective Receipts / verify --explain / Nutrition Label patterns.
3. Triage: is this a deficit to fix, a feature to design, or noise to ignore?
4. Persist a memory entry under `feedback_*` if the finding is recurring or load-bearing.

### 7.5 Monthly (heavy touch)

1. Re-read this charter end-to-end.
2. Re-read `PRODUCT_DELIBERATION_2026-05-14.md` and `ZENITH_FRAMING_2026-05-16.md`.
3. Check for drift between memory and reality (versions, claims, statuses).
4. Look for "soon" / "deferred" / "pending" items that have aged into "abandoned" — explicitly close them or re-justify them.
5. Look for stale claims, stale `T1x` / `until X` references, stale docstrings.
6. Verify the live demo trust loop end-to-end (Python ↔ Worker receipt round-trip).
7. Confirm the proof-boundary discipline still holds — sweep for "guarantee" / overclaim words in any doc touched this month.

### 7.6 On external pull

Inbound from a journalist, researcher, agent-builder, enterprise, standards body, or potential customer:

1. **Respond fast.** External pull is the rarest and most valuable signal.
2. Capture the request shape in memory.
3. If the request maps to a planned-but-not-shipped concept (Perspective Receipts, Trust Profiles, etc.), this is the moment to ship.
4. If the request is for a substrate feature the substrate already serves, surface the existing surface and document the gap (the request is evidence that documentation is the bottleneck, not features).

## 8. How this document changes

This charter is durable but not immutable. It changes when:

- A grant signal materially shifts strategy (§3, §4).
- A dogfood finding rules out a hypothesis named in §3.
- An external pull rewires the wedge ICP.
- Three months pass without re-examination (§7.5 monthly loop).

It does NOT change when:

- A new clever framing arrives (the deliberation and zenith artifacts already absorbed two of those).
- The user feels uncertain about direction. The charter's job is to be the stable thing during uncertainty.
- A vision document from elsewhere proposes a bigger destination. That goes into a new framing artifact; the charter only changes if signal proves the framing is the new reality.

## 9. Operating principles (short list, always applied)

These are the rules embedded across the project's memory, surfaced here in one place:

- **Buyer-or-dream filter** — every PR must serve a named buyer, a funder commitment, or a dream element. Math-natural work is rejected.
- **Overdeliver directive** — make funder promises true; adversarial-reviewer test on every change.
- **Carmack-honest framing** — the grants don't ship SUM; the work ships SUM. The work is what converts SUM from "strong repo + good demo" into "a product a stranger pays for."
- **Scope-before-signal** — pick the smallest shippable set that doesn't pre-commit to which budget you're building against.
- **Features-before-monetization** — the substrate must do what it claims before the marketing matters.
- **Three-runtime byte-symmetry** — the cross-runtime trust triangle is a load-bearing claim; no change ships without K1–K4 + A1–A6 green.
- **Proof-boundary discipline** — every claim tagged `provable` / `certified` / `empirical-benchmark` / `expert-opinion`. No conflation.
